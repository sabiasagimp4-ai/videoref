# Eagle近接化 残り3機能 設計（スマートフォルダ / D&D取り込み / 重複検出）

## 背景・目的

複数ライブラリ対応（v1.2.0）に続き、Eagle近接化の残り3機能を実装する。
- スマートフォルダ（保存済み検索）
- ドラッグ&ドロップ取り込み
- 類似画像/重複検出

UI/コメント/レーティングの近接化については、調査の結果 **videorefは既にEagle相当の構成を持っている** ことが判明したため、新規実装ではなく差分の微調整のみ行う（後述）。

## 調査結果サマリー（Eagle実機能）

| 機能 | Eagleの実際の挙動 |
|---|---|
| スマートフォルダ | タグ/評価/色/名前/拡張子/寸法/日付/ソースURL条件で**動的に**再評価される保存検索。スマートフォルダ内に別スマートフォルダをネスト可能 |
| D&D取り込み | OSファイラーからD&D。ブラウザ拡張経由でWebページの画像をD&D保存（元URL記録）。バッチ保存対応 |
| 重複検出 | **完全一致のみ**判定（フォーマット/サイズが違えば重複としない）。グループ表示→メタデータ保持マージ→一括操作。音声はMD5明記、動画の対応は不明 |
| Notes/Rating/Tags/Color | インスペクタにタイトル・★1-5評価（テンキー1-5で直接設定）・タグ・**Notes**欄・カラースウォッチ |

## 既存コードとのギャップ分析

videorefの現状（確認済み）:
- `state.inspectorMeta = { tags, note, color, rating, url }` — Eagleの Notes/Rating/Tags/Color/SourceURL に**概念的に全対応済み**
- 評価のテンキー1-5ショートカットは`public/app.js:1728`付近で**既に実装済み**
- `collections`（`server.js:777` `loadCollections`/`saveCollections`）は**静的なitems配列**のみ — スマートフォルダ（動的フィルタ）は未実装
- ドラッグ&ドロップによるファイル取り込みは**未実装**（`drop`/`dragover`イベントリスナーがコード中に存在しない）
- 重複検出は**未実装**（ハッシュ計算・MD5等のコードが存在しない）

→ 結論: UI/Notes/Rating差分の新規実装は不要。残り3機能に集中する。

## 機能A: スマートフォルダ

### スコープ
含む: タグ・評価(以上)・色・拡張子・名前部分一致・追加日範囲・ソースURL部分一致によるAND条件の保存検索。既存コレクションのサイドバー位置に統合表示。
含まない: スマートフォルダのネスト（Eagle機能だが現状videorefにはフォルダのネスト構造自体が無いため対象外）、OR条件（v1はAND固定。Eagle側の正確なAND/OR UIが調査で確認できなかったため、シンプルなAND固定から始める）。

### データモデル
`collections.json`の各エントリに`type`を追加（既存データは`type`省略=`manual`として後方互換）:
```json
{
  "id": "col_xxx",
  "name": "高評価のRTA",
  "type": "smart",
  "rules": {
    "tags": ["RTA"],
    "ratingMin": 4,
    "color": null,
    "exts": [".mp4"],
    "nameContains": "",
    "dateFrom": null,
    "dateTo": null,
    "urlContains": ""
  }
}
```
`type: "manual"`（既存）は`items`配列を使い続ける。`type: "smart"`は`items`を持たない。

### 判定ロジック（フロントエンド・クライアントサイド）
既存のタグ/フォルダフィルタと同様、`state.files`（フル一覧、タグ・評価・色・url・mtimeを含む）に対してクライアント側でフィルタを実行する。バックエンド変更不要（既存`/api/files`のレスポンスで十分）。`applyFilters()`の`activeNav === 'collection'`分岐で、対象コレクションが`type === 'smart'`なら`items`参照ではなく`rules`によるフィルタ関数を通す。

### UI
- コレクション一覧（サイドバー）に表示する際、スマートフォルダは虫眼鏡アイコン等で通常コレクションと区別
- 「+ スマートフォルダ」ボタンを新設（既存「+ コレクション」ボタンの隣）→ルール編集モーダル（タグ複数選択チェックボックス、評価以上のスライダー/セレクト、色スワッチ、拡張子チェックボックス、名前/URL部分一致テキスト欄、追加日範囲）
- 既存スマートフォルダの編集も同モーダルを再利用

## 機能B: ドラッグ&ドロップ取り込み

### スコープ
含む: OSファイラーからvideorefウィンドウへのファイル/フォルダD&Dでライブラリへインポート。複数ファイル同時対応。進行表示。
含まない: ブラウザ拡張機能経由のWeb画像保存（別アプリ＝ブラウザ拡張の新規開発が必要なため、別サブプロジェクトとして切り出す）。

### 技術的制約（重要）
Electron 32以降、セキュリティ上の理由で`File.path`がレンダラーから直接取得できなくなった（本プロジェクトはElectron 41）。`webUtils.getPathForFile(file)`をpreload経由で呼ぶ必要がある。

`preload.js`に追加:
```js
const { contextBridge, ipcRenderer, webUtils } = require('electron');
// ...
getPathForFile: (file) => webUtils.getPathForFile(file),
```

### フロー
1. `public/app.js`: `#gallery`（またはウィンドウ全体）に`dragover`(preventDefault)・`drop`イベントを追加
2. `drop`イベントで`e.dataTransfer.files`から`window.electronAPI.getPathForFile(file)`で各ファイルの絶対パスを取得
3. フォルダがドロップされた場合の扱い: 第一段階ではファイルのみ対応（フォルダ丸ごとは`scanDir`を再帰的に呼ぶ必要があり複雑化するため、最初のトレースではスコープ外とし、TODOとして明記）
4. パス配列を`POST /api/import`に送信
5. ドロップ直後、確認ミニダイアログで「コピー」「移動」をユーザーが選択（決定: コピー/移動を選べるようにする。デフォルトはコピー）
6. サーバー側（新規エンドポイント `POST /api/import { paths, mode: 'copy'|'move' }`）: 各パスについて対応拡張子(`VIDEO_EXTS`/`IMAGE_EXTS`/`AUDIO_EXTS`)かチェックし、`LIBRARY_PATH`直下へ`mode`に応じて`fs.copyFileSync`または`fs.renameSync`（同名ファイルが存在する場合は`(1)`等のサフィックスを付与。`renameSync`がドライブ越え(EXDEV)で失敗した場合は既存`moveIfExists`と同様にコピー+削除へフォールバック）。完了後`scanDir`差分検出→`startBackgroundThumbGen`
7. 既存の`downloadJobs`と同様のジョブパターンで進行状況を返す（大量ファイル・大容量動画のコピー/移動は時間がかかるため、同期レスポンスではなくジョブID＋ポーリングにする）

### 確定事項
コピー/移動はドロップ時にユーザーが毎回選択する（決定済み）。デフォルト選択はコピー（データ消失リスクが低い方）。

## 機能C: 重複検出（完全一致のみ、Eagle仕様に準拠）

### スコープ
含む: ライブラリ内の完全一致ファイルの検出（ファイルサイズ＋ハッシュ）、グループ表示UI、メタデータ保持マージ、一括削除。
含まない: 視覚的な類似画像検出（perceptual hash等） — Eagle本体機能か不確実なため対象外。動画特有の「内容は同じだがコーデック/解像度が違う」類似検出も対象外（Eagle自身も完全一致のみと明記）。

### アルゴリズム
- 第一段階: ファイルサイズでグループ化（コストが低い）
- 同サイズグループ内のみSHA-256ハッシュを計算して完全一致を確定（全ファイルハッシュ化はコスト高なので避ける）
- 動画ファイルが多い前提のため、ハッシュ計算はバックグラウンドジョブ化必須（既存`thumbQueue`同様のキュー方式）

### API
- `POST /api/duplicates/scan` — スキャン開始（ジョブID返却、既存download/thumbジョブと同パターン）
- `GET /api/duplicates/scan/:jobId` — 進行状況取得
- `GET /api/duplicates` — 直近のスキャン結果（グループ配列: `[{ hash, files: [id, id, ...] }]`）
- `POST /api/duplicates/merge` — `{ keepId, removeIds: [...] }` 指定で、`removeIds`をゴミ箱へ移動しつつ、`keepId`のメタデータに`removeIds`側のタグ等をマージ（Eagle仕様: メタデータ保持マージ）

### UI
新規ナビ項目「重複ファイル」（ゴミ箱と同様にサイドバー下部）→グループごとにカード並列表示、グループ内で「残すファイル」を選択→マージ実行。

## 着手順（推奨）

1. **スマートフォルダ** — バックエンド変更ゼロで実装可能、既存コレクションUIの拡張のみ。リスク最小・効果が分かりやすい
2. **ドラッグ&ドロップ取り込み** — `webUtils`の技術検証が必要だが、既存ダウンロードジョブのパターンを再利用できる
3. **重複検出** — ハッシュ計算ジョブ・新規UI画面が必要で最も工数が大きい。動画ハッシュ計算の実時間コストも検証が必要

## 決定事項（ユーザー確認済み）

1. ドラッグ&ドロップ時のコピー/移動はドロップごとにユーザーが選択（デフォルトはコピー）
2. スマートフォルダはv1ではAND条件固定
3. 重複検出はEagle同様「完全一致のみ」（ファイルサイズ+ハッシュ）。視覚的類似検出は対象外（将来の別サブプロジェクト候補）
